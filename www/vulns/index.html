API Bruteforce/Information Disclosure: The Quigley

Don't get scared now! We're just getting going.

You won't be able to see this next vulnerability immediately; you gotta go find it. So take a deep breath, and plunge!

You're currently in the /vulns/ directory. There is an API that is accessible from your current directory, but you have to go find it.

Break out your best directory busting tools and go hunting for some APIs!

The API is not immediately accessible from the /vulns/ directory. In fact, it's a few directories down. APIs are often nested inside of many sub-directories. So you'll need to drill down a little bit to find it.

Try using dirb: dirb http://[IP]/vulns/

When you dirbust the /vulns/ directory, dirb will find a number of sub-directories and recursively search inside of them.

The API can be found by visiting/CURLing /vulns/api/v1/monitoring/accounts/

The API responds to a basic HTTP request a certain way. If you just visit or CURL the page by default, you are making a GET request to this REST API. If you review the source code of the API, you can see that it can also handle a POST request to add users to the database if it is given certain parameters.

So, if the author of this API programmed it to handle GET requests and POST requests that have paramerters, what else could we do to poke at the API?

Enter: curl -X POST http://[IP]/vulns/api/v1/monitoring/accounts/ to POST to the API with no parameters. This returns something unexpected!
